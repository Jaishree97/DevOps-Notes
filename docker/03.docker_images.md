# Docker Images

## What is a Docker Image?

A **Docker image** is a **read-only template** used to create Docker containers.

It packages everything an application needs to run:

- Application Code
- Dependencies
- Libraries
- Runtime
- Configuration Files

> **Image** = Blueprint (Template)  
> **Container** = Running Instance

### Key Characteristics

- Read-only
- Portable
- Reusable
- Built from a Dockerfile
- Composed of multiple layers

---

## Docker Image Workflow

```text
Dockerfile
     │
docker build
     │
Docker Image
     │
docker run
     │
Docker Container
```

---

## How Images Are Created

Docker images are built using a **Dockerfile**.

A Dockerfile is a text file containing instructions that Docker executes **from top to bottom** to create an image.

Build an image using:

```bash
docker build -t image-name .
```

Example:

```bash
docker build -t myapp:v1 .
```

---

# Docker Tags

## What is a Docker Tag?

A **Docker tag** is a label that identifies a specific version of an image.

A tag **does not create a new image**. It simply creates another reference to the same image.

### Common Uses

- Versioning
- Environment management
- Image identification
- Preparing images for Docker Hub

### Syntax

```bash
docker tag SOURCE_IMAGE[:TAG] TARGET_IMAGE[:TAG]
```

### Example

```bash
docker tag myapp:v1 myapp:latest
```

### Tag for Docker Hub

```bash
docker tag myapp:v1 username/myapp:v1
```

Push the image:

```bash
docker push username/myapp:v1
```

# Dockerfile

## What is a Dockerfile?

A **Dockerfile** is a text file that contains instructions used to automatically build a Docker image.

Docker executes these instructions **sequentially from top to bottom**.

---

# Dockerfile Instructions

## FROM

Specifies the **base image** used to build the application.

Every Dockerfile begins with a **FROM** instruction.

```dockerfile
FROM ubuntu:22.04
```

---

## WORKDIR

Sets the working directory inside the image.

If the directory does not exist, Docker creates it automatically.

```dockerfile
WORKDIR /app
```

---

## COPY

Copies files and directories from the local machine into the image.

### Syntax

```dockerfile
COPY <source> <destination>
```

### Example

```dockerfile
COPY package.json /app/
COPY . /app
```

Use **COPY** for most file-copy operations.

---

## ADD

Works similarly to **COPY** but provides additional features.

It can:

- Copy local files
- Extract compressed archives automatically
- Download files from URLs

Example:

```dockerfile
ADD app.tar.gz /app/
```

> **Best Practice:** Prefer **COPY** unless you specifically need ADD's additional features.

---

## RUN

Executes commands **during the image build process**.

Common uses:

- Install packages
- Update the operating system
- Build the application

Example:

```dockerfile
RUN apt update && apt install -y curl
```

> Every **RUN** instruction creates a new image layer.

---

## ENV

Defines environment variables that are available inside the container.

```dockerfile
ENV NODE_ENV=production
```

---

## ARG

Defines variables available **only during the image build**.

```dockerfile
ARG VERSION=1.0
```

Unlike **ENV**, ARG values are **not available** after the container starts.

---

## EXPOSE

Documents the port on which the application listens.

```dockerfile
EXPOSE 3000
```

> **Note:** `EXPOSE` does **not** publish the port.  
> Use `docker run -p HOST_PORT:CONTAINER_PORT` to make the application accessible.

Example:

```bash
docker run -p 3000:3000 myapp
```

---

## CMD

Specifies the **default command** executed when a container starts.

```dockerfile
CMD ["node","app.js"]
```

The CMD instruction can be overridden when running the container.

---

## ENTRYPOINT

Specifies the **main executable** that always runs when the container starts.

```dockerfile
ENTRYPOINT ["node"]
```

Runtime arguments are automatically appended.

### CMD vs ENTRYPOINT

| ENTRYPOINT | CMD |
|------------|-----|
| Main executable | Default command or arguments |
| Usually fixed | Easily overridden |
| Defines container behavior | Provides defaults |

They are often used together.

---

## USER

Specifies which user runs processes inside the container.

```dockerfile
USER node
```

> **Best Practice:** Avoid running containers as the root user whenever possible.

---

## LABEL

Adds metadata to an image.

```dockerfile
LABEL version="1.0"
LABEL maintainer="admin@example.com"
```

Labels help organize and document images.

---

# Docker Image Layers

Every Dockerfile instruction creates a **read-only image layer**.

Example:

```dockerfile
FROM ubuntu:22.04

RUN apt update

COPY . /app
```

### Layer Structure

```text
Layer 3 → COPY . /app
Layer 2 → RUN apt update
Layer 1 → FROM ubuntu:22.04
```

Think of layers as **stacked snapshots**.

### Benefits of Layers

- Faster builds
- Storage efficiency
- Shared between images
- Faster image downloads
- Efficient caching

---

# Docker Build Cache

Docker builds images **one instruction at a time**.

If an instruction and its dependencies have not changed, Docker reuses the cached layer instead of rebuilding it.

## Good Dockerfile Order

```dockerfile
COPY package.json .

RUN npm install

COPY . .
```

Only dependency changes trigger a new installation.

---

## Bad Dockerfile Order

```dockerfile
COPY . .

RUN npm install
```

Any source code change forces Docker to reinstall all dependencies.

> **Best Practice:** Place instructions that change less frequently near the top of the Dockerfile.

---

# Best Practices

- Use official base images whenever possible.
- Keep images as small as possible.
- Prefer **COPY** over **ADD** unless ADD's extra features are required.
- Combine related RUN commands to reduce image layers.
- Order Dockerfile instructions to maximize build cache.
- Avoid running applications as the root user.
- Tag images with meaningful versions instead of relying only on `latest`.

---

# Key Takeaways

- Docker images are **read-only templates** used to create containers.
- Images are built from a **Dockerfile**.
- Every Dockerfile instruction creates an image layer.
- Docker caches unchanged layers to speed up future builds.
- Proper instruction order significantly improves build performance.
- Image tags help manage different versions of the same image.
- Following Docker best practices results in smaller, faster, and more secure images.
