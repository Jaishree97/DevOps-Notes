# Docker Compose

## What is Docker Compose?

Docker Compose is a tool used to define and manage **multi-container Docker applications** using a **docker-compose.yml** file.

It allows you to:

- Define multiple services.
- Configure networking.
- Manage persistent storage with volumes.
- Set environment variables.
- Start and stop the entire application with a single command.

---

## Why Docker Compose?

- Simplifies multi-container application management.
- Automatically creates a network for services.
- Supports persistent data using volumes.
- Makes application configuration easier.
- Ideal for local development and testing.

---

# docker-compose.yml

## What is docker-compose.yml?

A **docker-compose.yml** file defines the services, networks, volumes, and configuration required to run a multi-container application.

### Basic Structure

```yaml
version: "3.9"

services:
  web:
    image: nginx
    ports:
      - "8080:80"

  db:
    image: mysql

volumes:
  db_data:
```

---

## Main Sections

- **services** → Defines application containers.
- **volumes** → Stores persistent data.
- **networks** → Defines custom networks (optional).
- **environment** → Stores configuration values.

# Key Concepts

## Services

A **service** represents a single container.

Services are defined under the `services` section.

Each service:

- Runs in its own container.
- Can communicate with other services.
- Can have its own ports, volumes, and environment variables.

---

## Networking

Docker Compose automatically creates a **default network**.

All services connected to the same Compose project can communicate using their **service names**.

### Example

```yaml
services:
  wordpress:
    image: wordpress
    environment:
      WORDPRESS_DB_HOST: db

  db:
    image: mysql
```

Here, **db** acts as the hostname for the MySQL service.

---

## Volumes

Docker Compose supports **named volumes** for persistent data.

### Example

```yaml
services:
  db:
    image: mysql
    volumes:
      - db_data:/var/lib/mysql

volumes:
  db_data:
```

### Why Use Volumes?

- Preserve data after container removal.
- Share data between containers.
- Recommended for databases and production workloads.

---

## Environment Variables

Environment variables allow configuration values to be passed into containers.

### Option 1: Inside the Compose File

```yaml
environment:
  MYSQL_ROOT_PASSWORD: example
```

---

### Option 2: Using a `.env` File (Recommended)

**.env**

```text
MYSQL_ROOT_PASSWORD=example
```

**docker-compose.yml**

```yaml
environment:
  MYSQL_ROOT_PASSWORD: ${MYSQL_ROOT_PASSWORD}
```

### Advantages

- Cleaner configuration
- Easier environment management
- Better security
- Production-ready configuration

---

## depends_on

The `depends_on` option controls the startup order of services.

### Example

```yaml
services:
  web:
    depends_on:
      - db

  db:
    image: mysql
```

The **db** service starts before the **web** service.

> **Note:** `depends_on` controls only the startup order. It does **not** wait until the dependent service is fully ready.

# Essential Docker Compose Commands

## Start Services

```bash
docker compose up
```

---

## Start in Detached Mode

```bash
docker compose up -d
```

---

## Stop Services

```bash
docker compose stop
```

---

## Remove Containers and Networks

```bash
docker compose down
```

---

## Remove Containers, Networks, and Volumes

```bash
docker compose down -v
```

---

## View Running Services

```bash
docker compose ps
```

---

## View Logs

```bash
docker compose logs
```

---

## View Logs for a Specific Service

```bash
docker compose logs db
```

---

## Rebuild Images

```bash
docker compose up --build
```

# Single vs Multi-Container Architecture

## Single Container

One container runs:

- Web Server
- Application
- Database

### Limitations

- Difficult to scale.
- Harder to maintain.
- Less flexible.

---

## Multi-Container

Each service runs in its own container.

Example:

- WordPress
- MySQL

### Advantages

- Separation of concerns
- Better scalability
- Easier debugging
- Independent updates
- Improved maintainability

---

# Example: WordPress + MySQL

### Requirements

- Two services
- Automatic networking
- Named volume for MySQL
- WordPress connects using the **db** service name

### Persistence Test

```bash
docker compose down

docker compose up
```

The database remains available because the named volume preserves the data.

# Docker Compose V1 vs V2

### Compose V1

```bash
docker-compose
```

### Compose V2

```bash

```

Modern Docker installations use **Docker Compose V2**.

# Quick Summary

- Docker Compose is a multi-container orchestration tool.
- Uses a `docker-compose.yml` file.
- Services communicate using service names.
- Docker automatically creates a default network.
- Named volumes provide persistent storage.
- `.env` files simplify configuration management.
- `depends_on` controls service startup order.
- `docker compose up -d` starts services in the background.
- `docker compose down` removes containers and networks.
- `docker compose down -v` also removes named volumes.

---

# Key Takeaways

- Docker Compose simplifies running multiple containers.
- Each service runs independently.
- Automatic networking eliminates manual container linking.
- Named volumes keep application data persistent.
- Environment variables make applications portable.
- Docker Compose is the preferred tool for local development and testing.
