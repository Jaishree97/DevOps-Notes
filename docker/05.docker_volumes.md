# Docker Volumes

## Why Do We Need Docker Volumes?

Docker containers are **ephemeral** by default.

When a container is removed:

- Its writable layer is deleted.
- All data stored inside the container is lost.

To persist data beyond the container lifecycle, Docker provides **Docker Volumes**.

---

## What are Docker Volumes?

Docker Volumes are persistent storage managed by Docker.

They:

- Are managed by Docker.
- Exist outside the container's filesystem.
- Persist even after a container is removed.
- Can be shared between multiple containers.
- Are recommended for databases and production workloads.

---

## Where are Volumes Stored?

Docker stores volumes outside the container's writable layer.

On Linux, volumes are typically stored under:

```text
/var/lib/docker/volumes/
```

Docker manages this location automatically, so users usually do not need to access it directly.

---

## How Docker Volumes Work

```text
        Application
             │
             ▼
      Docker Container
             │
             ▼
       Docker Volume
             │
             ▼
        Host Storage
```

Even if the container is deleted, the data remains stored in the Docker Volume.

---

# Volume Commands

## Create a Volume

```bash
docker volume create my-volume
```

---

## List Volumes

```bash
docker volume ls
```

---

## Inspect a Volume

```bash
docker volume inspect my-volume
```

---

## Remove a Volume

```bash
docker volume rm my-volume
```

---

# Example: MySQL with Persistent Storage

### Step 1: Create a Volume

```bash
docker volume create db-data
```

---

### Step 2: Run MySQL Using the Volume

```bash
docker run -d \
  --name sql-db \
  -e MYSQL_ROOT_PASSWORD=test@123 \
  -v db-data:/var/lib/mysql \
  -p 3306:3306 \
  mysql
```

---

### Step 3: Create a Database

Enter the MySQL container.

```bash
docker exec -it sql-db mysql -u root -p
```

Create a database.

```sql
CREATE DATABASE myapp;
```

---

### Step 4: Remove the Container

```bash
docker stop sql-db

docker rm sql-db
```

---

### Step 5: Recreate the Container

```bash
docker run -d \
  --name sql-db \
  -e MYSQL_ROOT_PASSWORD=test@123 \
  -v db-data:/var/lib/mysql \
  -p 3306:3306 \
  mysql
```

The **myapp** database still exists because the data is stored in the Docker Volume.

# Bind Mounts

## What are Bind Mounts?

Bind Mounts map a file or directory from the host machine directly into a container.

They:

- Are **not managed** by Docker.
- Depend on the host filesystem.
- Reflect file changes immediately.
- Are commonly used during development.

---

## Example: Nginx with Bind Mount

### Step 1: Create a Project Directory

```bash
mkdir my-website

cd my-website
```

---

### Step 2: Create an HTML File

```bash
echo "<h1>Hello from Bind Mount</h1>" > index.html
```

---

### Step 3: Run the Container

```bash
docker run -d \
  --name nginx-dev \
  -p 8080:80 \
  -v $(pwd):/usr/share/nginx/html \
  nginx
```

---

### Step 4: Open the Application

```
http://localhost:8080
```

Edit **index.html** and refresh the browser.

The changes appear immediately without rebuilding the container.

---

# Volumes vs Bind Mounts

| Feature | Docker Volumes | Bind Mounts |
|---------|----------------|-------------|
| Managed by Docker | Yes | No |
| Data Persistence | Yes | Yes |
| Host Filesystem Dependency | No | Yes |
| Best for Development | Limited | Yes |
| Best for Production | Yes | Limited |
| Best for Databases | Yes | No |

---

# When to Use What?

## Use Docker Volumes

- Database storage
- Production applications
- Persistent application data
- Shared storage between containers

---

## Use Bind Mounts

- Local development
- Source code synchronization
- Configuration files
- Live file editing

---

# Key Takeaways

- Containers lose their writable data when removed.
- Docker Volumes provide persistent storage managed by Docker.
- Volumes survive container removal and can be reused.
- Bind Mounts map host files directly into containers.
- Docker Volumes are recommended for production workloads.
- Bind Mounts are ideal for development because changes are reflected immediately.
