# Docker Image Optimization

## What is a Multi-Stage Build?

A **Multi-Stage Build** allows you to use multiple `FROM` instructions in a single Dockerfile.

Each stage has a specific purpose:

- **Stage 1** → Build the application
- **Stage 2** → Create the production image

Only the required application files are copied into the final image, resulting in a **smaller, cleaner, and more secure** production image.

---

## Before vs After

| Single-Stage Build | Multi-Stage Build |
|--------------------|------------------|
| Larger image size | Smaller image size |
| Includes build tools | Contains only production files |
| Slower deployments | Faster deployments |
| Larger attack surface | Improved security |
| More storage usage | Better storage efficiency |

---

## Why Use Multi-Stage Builds?

- Reduce image size
- Faster image downloads
- Faster deployments
- Better security
- Remove unnecessary build tools
- Create cleaner production images

---

## Example

```dockerfile
# ---------- Stage 1: Build ----------
FROM node:22 AS builder

WORKDIR /app

COPY package*.json ./

RUN npm ci

COPY . .

RUN npm run build

# ---------- Stage 2: Production ----------
FROM node:22-alpine

WORKDIR /app

COPY --from=builder /app/dist ./dist

CMD ["node","dist/app.js"]
```

The final image contains only the production application and its runtime.

---

# Lightweight Base Images

Using lightweight base images helps reduce image size and improve performance.

Example:

```dockerfile
FROM node:22-alpine
```

### Benefits

- Smaller image size
- Faster downloads
- Lower storage usage
- Reduced attack surface

---

# Distroless Images

## What are Distroless Images?

Distroless images contain **only the components required to run an application**.

They include:

- Application runtime
- Required libraries
- Essential system files

They do **not** include:

- Shell (`bash` or `sh`)
- Package manager
- Debugging tools

Example:

```dockerfile
FROM gcr.io/distroless/nodejs22-debian12
```

> **Note:** Distroless images do not include a shell, making them more secure but harder to debug.

---

## Why Use Distroless Images?

- Extremely small image size
- Better security
- Reduced attack surface
- Production-ready containers
- Fewer vulnerabilities

---

## Multi-Stage + Distroless Example

```dockerfile
# ---------- Stage 1: Build ----------
FROM node:22 AS builder

WORKDIR /app

COPY package*.json ./

RUN npm ci

COPY . .

RUN npm run build

# ---------- Stage 2: Production ----------
FROM gcr.io/distroless/nodejs22-debian12

WORKDIR /app

COPY --from=builder /app/dist ./dist

CMD ["dist/app.js"]
```

The final image contains:

- Production application
- Runtime only
- Required libraries
- No source code
- No build tools
- No package manager
- No shell

# Optimize Docker Build Cache

Docker builds images **one instruction at a time**.

If an instruction has not changed, Docker reuses the cached layer instead of rebuilding it.

## Good Dockerfile Order

```dockerfile
COPY package*.json ./

RUN npm ci

COPY . .
```

Only dependency changes trigger a new package installation.

---

## Bad Dockerfile Order

```dockerfile
COPY . .

RUN npm ci
```

Every source code change forces Docker to reinstall all dependencies.

> **Best Practice:** Copy dependency files before application source code to maximize cache reuse.

---

# Use a `.dockerignore` File

A `.dockerignore` file prevents unnecessary files from being sent to the Docker build context.

Example:

```text
node_modules
.git
.gitignore
README.md
Dockerfile
Dockerfile.*
npm-debug.log
```

### Benefits

- Faster builds
- Smaller build context
- Better cache performance
- Cleaner images

---

# Best Practices

- Use Multi-Stage Builds for production images.
- Prefer lightweight base images such as Alpine.
- Use Distroless images for maximum security.
- Keep images focused on a single application.
- Copy dependency files before application source code.
- Use `npm ci` instead of `npm install` in CI/CD pipelines.
- Use a `.dockerignore` file to reduce the build context.
- Minimize the number of image layers.
- Run containers as a non-root user whenever possible.
- Keep base images updated with the latest security patches.
- Tag images with meaningful versions instead of relying only on `latest`.

---

# Key Takeaways

- Multi-Stage Builds create smaller and cleaner production images.
- Lightweight base images reduce image size and improve performance.
- Distroless images provide better security by removing unnecessary components.
- Docker Build Cache speeds up image builds by reusing unchanged layers.
- A `.dockerignore` file reduces build time and improves cache efficiency.
- Following Docker best practices results in faster, smaller, and more secure container images.
